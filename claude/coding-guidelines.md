# Coding Guidelines（脚本辅助模式）

当确定需要编写 Python 等脚本辅助数据处理时，遵循以下规范。仅在编码模式下读取本文件。

---

## 1. 先思考，后编码
- 不要假设。不要隐藏困惑。呈现权衡方案。
- 如果不确定，提出疑问；如果存在多种解读，列出供选择。
- 如果存在更简单的方案，要提出。

## 2. 简洁至上
- 用最少的代码解决问题。不写任何推测性内容。
- 不添加任何未被要求的功能。
- 不因为只有一次调用就引入抽象层。
- 不为不可能发生的场景编写错误处理。

## 3. 精准修改
- 只触碰必须改动的部分。只清理你自己造成的烂摊子。
- 不要顺手"优化"相邻的代码、注释或格式。
- 遵循现有代码风格，即使你自己更常用另一种写法。
- 删除那些**因为你的改动**才变得无用的导入、变量或函数。
- 检验标准：每一行被改动的代码，都应能追溯到用户的原始需求。

## 4. 目标计划执行
- 定义成功标准。循环直至验证通过。
- "添加验证" → "先为无效输入编写测试，然后让测试通过"
- "修复这个 Bug" → "先写一个能复现该问题的测试，然后使其通过"

## 5. 先读后写
- 编写脚本或修改报表前，先通读和理解相邻代码、模板结构及关联文件。
- 避免因未读既有逻辑而引入重复函数、冲突公式或破坏现有数据流。

## 6. 安全与权限
- **绝不**访问或修改当前工作目录以外的文件，除非用户明确授权。
- 执行 `Shell` 命令前确认其安全性，避免破坏性操作（如 `rm -rf`、`del /s` 等）。
- 对文件写入操作保持谨慎，修改前确认路径正确。

## 7. 工具使用
- 优先使用内置工具操作文件，而非 Shell 命令。
- 多文件读取时尽量并行调用，提高效率。
- 处理大文件时，分块读取，避免一次性加载全部内容。

## 8. 调试与验证
- 修改代码后尽可能运行测试或验证脚本，确保变更有效。
- 如果测试失败，分析错误日志并修复，迭代到通过为止。
- 涉及 Excel 文件时，验证结构完整性和数据正确性（无重复、无时间冲突等）。
- **测试通过不等于结果正确**：验证必须检查实际数据行为（数值范围合理性、时间连续性、主键唯一性），而非仅验证"未报错"。

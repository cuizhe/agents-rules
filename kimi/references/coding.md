# Coding Guidelines (脚本辅助模式)

当确定需要编写 Python 等脚本辅助数据处理时，遵循以下规范：

## 1. 先思考，后编码
- 不要假设，不隐藏困惑。**呈现权衡方案**。
- 如果不确定，提出疑问；如果存在多种解读，列出供选择。
- 修改前阅读相关文件的导出、调用方和共用工具。

## 2. 简洁精准
- 用最少的代码解决问题。不写推测性内容，不添加未被要求的功能。
- 只触碰必须改动的部分，不"顺手优化"相邻代码。
- **配合既有惯例，即使你不同意：** snake_case 就 snake_case、既有风格就延续。
- 删除**因你的改动**才变得无用的导入、变量或函数。
- 不为不可能发生的场景编写错误处理。

## 3. AI 与代码的职责边界
- 把 AI 用于需要判断的任务（分类、起草、摘要、逻辑分析）。
- **确定性决策**（重试策略、路由、状态码、格式转换、数值计算）必须用明确、可预测的代码实现，不依赖 AI 推理。
- 遇到冲突的代码模式时，**明确选择其一**（优先较新、较有测试覆盖的），混合两种模式是最差选择。

## 4. 安全与工具使用
- **绝不**访问或修改当前工作目录以外的文件，除非用户明确授权。
- 优先使用内置工具（`ReadFile`、`WriteFile`、`StrReplaceFile`、`Glob`、`Grep`）操作文件，而非 Shell 命令。
- 多文件读取时并行调用，大文件使用 `line_offset` + `n_lines` 分块读取。
- 对 `WriteFile`、`StrReplaceFile` 保持谨慎，修改前确认路径正确。

## 5. 调试与验证
- 修改后运行测试或验证脚本，确保变更有效。
- 涉及 Excel 时，验证结构完整性和数据正确性（无重复、无时间冲突等）。
- **测试要验证"意图"，不只验证"行为"：** 能写一个"业务逻辑改变时会失败"的测试才算合格。

# Workflow (QA 主模式)

## 任务启动与上下文理解
- **读懂再动手：** 处理数据、报表或质量问题前，先确认数据来源、字段定义、统计口径和最新版本。读不懂就不要改。
- **AI 与确定性操作的边界：** 把 AI 用于需要判断的复杂任务（根因分析、AAR 总结、风险分类）；确定性操作（数据透视、公式计算、固定格式转换）优先用 Excel 公式或模板，不用 AI 做简单可规则化的事。

## 数据处理与报表
- 所有数据、报表、分析必须准确，来源可追溯。
- 偏好结构化、分步骤的工作方式，符合质量管理流程。
- 表格处理时，注意公式逻辑校验，避免数据错误导致决策偏差。
- 输出文档或报告时，格式规范、层次清晰，适合向上汇报或团队同步。
- 优先复用已有模板，保持团队一致性。

## 多步骤任务与 Checkpoint
- **阶段总结：** 每完成一个阶段，主动总结"做了什么、验证了什么、还剩什么"。无法清晰描述状态时，暂停并澄清，不盲目前进。
- **上下文预算：** 长对话存在上下文限制。单轮接近极限时，主动摘要并重启，不无声突破。

## 质量分析与回溯
- 遵循"现象 → 根因 → 措施 → 验证"的逻辑链。
- AAR：项目关键节点或问题闭环后的事后回顾。
- 质量回溯：针对严重或重复出现的问题，进行根因分析和预防措施制定。
- 月度收入/成本看护：跟踪实际发生与预算的偏差，预警盈利风险。

## 数据敏感度
- 涉及项目收入、成本、人天等数据时，注意保密和权限范围。

## Skill 使用规范（铁律）
- **当任务涉及已加载的 Skill 时，必须先检查该 Skill 自带的 `scripts/`、`assets/`、`references/` 目录，严格遵循 Skill 提供的工具和路径。**
- **Skill 的脚本、启动器、配置文件优先级高于任何通用解决方案。** 需要启动浏览器时，先查看 Skill 的 `scripts/` 下是否有 launcher，而不是自己写 shell 命令。
- 违反此规则的典型错误：忽略已有的 launcher，自行拼凑启动命令，导致路径错误、参数缺失、端口冲突。

## Kimi CLI 自身维护
- **更新 kimi-cli 必须使用 `uv tool upgrade kimi-cli`，绝对禁止使用 `pip install`。**
  - pip 会安装到 Node.js 默认路径，导致多版本并存，uv 无法覆盖。
  - 验证：`kimi --version` / `uv tool list | grep kimi`
- **Skills 管理架构**：`skills-manager` 桌面应用统一管理，唯一副本位于 `C:\Users\Administrator\.skills-manager\skills\`，各智能体通过**软连接**同步。修改即改统一仓库，Git 提交同步到 GitHub。
